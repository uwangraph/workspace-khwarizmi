# Dokumentasi Test Cases & Bug Report — Khwarizmi Workspace

Dokumen ini disusun untuk tim QA sebagai panduan pengujian manual pada **Khwarizmi Workspace** — platform manajemen kerja *mobile-first* milik PT. Alkhwarizmi Kreatif Produktif yang mencakup tiga modul utama: **Presensi Harian**, **Task Management**, dan **Chat Realtime**. Dokumen mencakup skenario pengujian untuk **Role User** (Karyawan) dan **Role Admin** (Administrator), beserta temuan bug kritis.

---

## 1. Standar Penulisan & Klasifikasi Tipe Pengujian

Seluruh skenario ditulis dengan pendekatan **BDD (Behavior-Driven Development)** menggunakan klausa *Given-When-Then* dan aktor **"As User"** untuk konsistensi.

### Klasifikasi Tipe Pengujian:
1. **Smoke Testing** — Fungsi inti paling krusial; gagal berarti aplikasi tidak bisa digunakan.
2. **Happy Flow** — Skenario ideal; semua input valid, alur berjalan sempurna tanpa hambatan.
3. **Positive** — Skenario sukses lainnya; fitur opsional, variasi input valid, interaksi UI/UX.
4. **Negative** — Skenario gagal; input salah, field kosong, atau pelanggaran *business rules*.
5. **Regressions** — Pengujian ulang fitur lama setelah ada perubahan sistem.
6. **Edge Case** — Skenario ekstrem atau tidak biasa yang berpotensi merusak logika sistem.

## 4. Temuan Bug Kritis — Client-Side Auto-Checkout

Selama analisis kode ditemukan sebuah *critical bug* arsitektur pada modul absensi. Sistem auto-checkout bergantung sepenuhnya pada sisi *frontend* (dipicu oleh `loadData()`), sehingga tidak pernah berjalan jika pengguna tidak membuka halaman absensi tepat waktu.

### Bug Report: Auto-Checkout Tidak Berjalan Saat Pengguna Tidak Membuka Halaman
* **Bug ID**: BUG-ATTENDANCE-001
* **Severity**: Major / Critical
* **Component**: `src/routes/absensi/+page.svelte` → fungsi `runAutoCheckout()` & `loadData()`
* **Status**: Open / FAIL

### Skenario Reproduksi:
1. **Given** Karyawan melakukan Clock In pada Sesi Pagi tanggal 17 Mei jam 08:30.
2. **When** Karyawan menutup aplikasi dan tidak membuka halaman `/absensi` lagi sepanjang hari.
3. **Then** Fungsi `runAutoCheckout()` yang seharusnya berjalan saat waktu melewati `autoCheckoutAt = '12:00'` **tidak pernah terpicu**, karena ia hanya dipanggil di dalam `loadData()` pada `onMount` — tidak ada *background worker* atau *server-side cron job*.
4. **And** Penalti lupa checkout (10 menit) ke tabel `attendance_penalties` juga tidak tercatat.
5. **And** Pada hari berikutnya (18 Mei), karyawan membuka halaman absensi. `loadData()` berjalan dan memanggil `runAutoCheckout()` yang memeriksa sesi dengan tanggal **kemarin** (`attendance.date = '2025-05-17'`).
6. **And** Fungsi ini mengisi `clock_out` untuk sesi kemarin dengan waktu auto-checkout kemarin — namun kondisi UI menggunakan `attendanceService.getTodayData()` sehingga **sesi kemarin tidak tampil**, namun state dari sesi yang terbuka satu hari sebelumnya bisa menyebabkan UI menganggap ada sesi aktif yang belum ditutup.
7. **And** Hasil akhirnya adalah **state yang ambigu**: backend memiliki sesi terbuka kemarin, frontend mencoba menutupnya hari ini, tapi sesi baru untuk hari ini (18 Mei) tidak bisa dimulai sampai kondisi kemarin beres.

### Kriteria Perbaikan (Acceptance Criteria untuk Developer):
* **Backend (Solusi Utama)**: Buat **Supabase Edge Function** terjadwal via Supabase cron (atau pg_cron) yang berjalan otomatis di `23:59` setiap harinya. Fungsi ini memindai tabel `attendance` untuk menemukan baris yang `clock_in` terisi, `clock_out` masih `NULL`, dan `date` adalah hari ini — kemudian mengisi `clock_out` secara otomatis dan mencatat penalti di `attendance_penalties`. Dengan demikian, sesi **pasti** selalu tertutup sebelum pergantian hari, tanpa bergantung pada user membuka aplikasi.
* **Frontend (Solusi Pelengkap)**: Di awal `loadData()`, tambahkan pengecekan: jika ada sesi di `attendance` dengan `date != today` yang masih `clock_out = null`, tampilkan pesan informatif *"Sesi dari tanggal [X] belum terselesaikan dan sudah otomatis dicatat sebagai lupa checkout."* — sehingga pengguna tidak bingung dengan state yang tidak konsisten.
